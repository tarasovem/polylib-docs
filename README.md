# Что такое Polylib

Polylib — минималистичная библиотека для разработки Web Components с декларативными шаблонами, реактивными свойствами и нативной моделью компонентов платформы.

В основе библиотеки — подход, при котором компонент остается изолированной единицей интерфейса: состояние хранится в свойствах, а интерфейс описывается шаблоном и автоматически синхронизируется с данными.

## Что можно строить

С библиотекой удобно создавать:

- переиспользуемые UI-компоненты;
- библиотеки компонентов и дизайн-системы;
- встраиваемые виджеты для проектов с разным стеком;
- интерфейсы, где важна нативная совместимость Web Components.

## Почему выбирают эту библиотеку

- **Нативный фундамент**: каждый компонент — стандартный `Custom Element`.
- **Декларативный UI**: шаблон описывает, как интерфейс должен выглядеть при текущем состоянии.
- **Реактивные свойства**: свойства компонента выступают как публичный API и единый источник состояния.
- **Предсказуемая стилизация**: `css`, `:host`, `::part`, `::slotted`, CSS custom properties.
- **Низкий порог старта**: можно начать с `index.html` без обязательной сборки.

## Как работает компонент

Рабочий цикл компонента:

1. Вы объявляете реактивные свойства.
2. Шаблон читает значения свойств и формирует UI.
3. При изменении свойств компонент обновляет интерфейс в соответствии с шаблоном.

Рекомендация: обновляйте состояние через свойства и методы компонента, а не через прямые манипуляции DOM.

## Минимальный пример

```js
import { PlElement, html } from "polylib";

class HelloCard extends PlElement {
  static properties = {
    name: {
      type: String,
      value: "World",
    },
  };

  static template = html`
    <h2>Hello, [[name]]</h2>
  `;
}

customElements.define("hello-card", HelloCard);
```

```html
<hello-card></hello-card>
```

## Требования

- базовое знание HTML, CSS, JavaScript;
- понимание Web Components на уровне общих принципов.

## Когда Polylib подходит лучше всего

- когда вы строите дизайн-систему на Web Components;
- когда нужны переиспользуемые компоненты между проектами;
- когда важны инкапсуляция, контролируемая стилизация и интеграция без привязки к фреймворку.

## Ключевые концепции

- **Свойства**: публичный API и внутреннее состояние компонента.
- **Шаблоны**: декларативное описание DOM в зависимости от данных.
- **События**: обмен действиями между компонентами и внешним окружением.
- **`slots` и Shadow DOM**: композиция интерфейса и инкапсуляция реализации.
- **`parts` и CSS custom properties**: безопасная внешняя кастомизация стилей.
